# autoevolve — автономное совершенствование стратегии DatsSol

## Описание

Автономный цикл эволюции бота. Claude итеративно модифицирует стратегию, прогоняет versus-турнир против снапшотов предыдущих версий, оценивает результат и решает — оставить или откатить. Снапшоты успешных версий сохраняются и участвуют в последующих матчах как оппоненты.

## Настройка

1. **Создать ветку**: `git checkout -b autoevolve/<tag>` (тег = дата, напр. `apr17`)
2. **Прочитать файлы** (полный контекст):
   - `strategy/bots/current.py` — **единственный файл который ты модифицируешь**
   - `strategy/core/rules.py` — движок симуляции (READ ONLY)
   - `strategy/core/state.py` — модель мира (READ ONLY)
   - `strategy/core/mapgen.py` — генерация карт (READ ONLY)
   - `strategy/base.py` — интерфейс бота (READ ONLY)
   - `api/models.py` — типы данных (READ ONLY)
3. **Инициализировать results.tsv** с заголовком
4. **Запустить baseline** — соло-турнир текущего бота
5. **Создать первый снапшот**: `make snapshot` — это baseline-оппонент

## Правила

### Что МОЖНО менять
- `strategy/bots/current.py` — **только этот файл**. Всё fair game: алгоритмы, эвристики, скоринг, приоритеты апгрейдов, логика стройки/ремонта/диверсий, планирование маршрутов, защита ЦУ.

### Что НЕЛЬЗЯ менять
- Всё в `strategy/core/` — движок, модель мира, генерация карт
- `strategy/base.py`, `api/models.py` — интерфейсы
- `strategy/runner.py`, `strategy/tournament.py`, `strategy/analyzer.py`, `strategy/snapshot.py` — инфраструктура
- Файлы в `strategy/bots/snapshots/` — это замороженные оппоненты

## Оценка: два режима

### Соло-турнир (быстрая проверка)
```bash
rm -f strategy/experiments/runs.csv
uv run python -m strategy.tournament --seeds 20 > run.log 2>&1
tail -5 run.log
```
Метрика: **средний score по 20 сидам**. Быстрый (~1 сек). Используй для первичной оценки.

### Versus-турнир (реальная проверка)
```bash
rm -f strategy/experiments/runs.csv
uv run python -m strategy.tournament --versus --seeds 20 --bots current,v001,v002 > run.log 2>&1
tail -10 run.log
```
Метрика: **win-rate против снапшотов** + **средний score в мультиплеере**. Все боты играют на одной карте одновременно — строят, мешают друг другу, конкурируют за клетки. Используй для финальной валидации.

### Какой режим когда
- **Соло** — когда меняешь экспансию, апгрейды, стройку (90% случаев)
- **Versus** — когда добавляешь диверсии, защиту от врагов, или после каждых 3-5 успешных соло-итераций

## Логирование

Файл `results.tsv` (tab-separated):

```
commit	mean_score	vs_winrate	status	description
```

- commit: short hash (7 символов)
- mean_score: средний score по соло-турниру из 20 сидов
- vs_winrate: win-rate в versus (0 если не проверялось, или 60.0, 80.0 и т.д.)
- status: `keep`, `discard`, `crash`
- description: что пробовали

Пример:
```
commit	mean_score	vs_winrate	status	description
a1b2c3d	56914	0	keep	baseline
b2c3d4e	62100	80.0	keep	prioritize reinforced cells + snapshot v002
c3d4e5f	54000	0	discard	aggressive sabotage (worse solo)
d4e5f6g	0	0	crash	infinite loop in pathfinding
```

## Цикл экспериментов

LOOP FOREVER:

### Фаза 1: Гипотеза
1. Посмотри текущее состояние: ветка, коммит, последние результаты в results.tsv
2. Сформулируй гипотезу: что изменить и почему

### Фаза 2: Реализация
3. Модифицируй `strategy/bots/current.py`
4. `git commit -am "описание изменения"`

### Фаза 3: Соло-проверка
5. `rm -f strategy/experiments/runs.csv`
6. `uv run python -m strategy.tournament --seeds 20 > run.log 2>&1`
7. `tail -5 run.log`
8. Если mean_score хуже baseline → **discard**, `git reset --hard HEAD~1`, перейди к шагу 1
9. Запиши в results.tsv

### Фаза 4: Versus-проверка (каждые 3-5 keep-итераций)
10. Собери список снапшотов: `uv run python -m strategy.snapshot list`
11. `rm -f strategy/experiments/runs.csv`
12. `uv run python -m strategy.tournament --versus --seeds 20 --bots current,v001,v002,... > run.log 2>&1`
13. `tail -10 run.log`
14. Если win-rate >= 55% → **keep**, иначе **discard**

### Фаза 5: Снапшот
15. При каждом **keep** после versus: `uv run python -m strategy.snapshot create`
16. Теперь этот снапшот будет оппонентом в будущих versus-матчах

### Фаза 6: Итерация
17. Если **keep** — оставляем коммит, идём к шагу 1
18. Если **discard** — `git reset --hard HEAD~1`, идём к шагу 1

## Управление снапшотами

```bash
make snapshot         # создать снапшот текущего current.py
make snapshots        # показать все снапшоты
make versus ARGS="--bots current,v001,v002 --seeds 20"  # versus-матч
```

Снапшоты создаются автоматически: копия `current.py` → `snapshots/vNNN.py` с переименованием класса. Реестр ботов `get_all_bots()` автоматически обнаруживает все снапшоты.

### Правила снапшотирования
- **Baseline** = v001 (первый снапшот перед началом цикла)
- **Каждый successful versus** → новый снапшот (v002, v003, ...)
- Не делай снапшот для каждого соло-keep — только для versus-валидированных
- В versus всегда включай v001 (baseline) + 2-3 последних снапшота

## Обработка ошибок
- Краш от бага — исправь и перезапусти
- Краш фундаментальный — откати, запиши crash
- Турнир дольше 30 секунд — что-то не так, убей и откати

## НИКОГДА НЕ ОСТАНАВЛИВАЙСЯ
После старта цикла НЕ спрашивай "продолжить?". Человек может спать. Работай до ручной остановки. Если кончились идеи — перечитай правила игры ниже, комбинируй подходы, пробуй радикальные изменения.

## Правила игры DatsSol (справка)

### Ключевые механики
- 600 ходов/раунд, 1 сек/ход
- Плантации связаны ортогонально с ЦУ (диагональ не считается)
- Каждая плантация: 1 действие/ход + автотерраформация
- Действия: строить, ремонтировать, диверсия, атака бобров
- Потеря ЦУ = всё уничтожено + штраф 5% очков
- Усиленные клетки (x%7==0 && y%7==0): 1.5x очков
- Плантация исчезает при 100% терраформации

### Параметры
- MHP=50, TS=5, CS=5, RS=5, DS=10, AR=2, SR=3
- Стройка: 50 HP до завершения, CS=5/ход → 10 ходов
- Терраформация: +5%/ход → 20 ходов до 100%
- Очки: 10 за 1% (обычная), 15 за 1% (усиленная)
- Лимит плантаций: 30 (+1 за апгрейд)
- Апгрейд каждые 30 ходов, макс 15

### Congestion (перегрузка)
Каждая следующая команда через одну выходную точку: CS/RS/SE/BE уменьшается на 1

### Апгрейды
- repair_power: RS +1 (макс 3)
- max_hp: MHP +10 (макс 5)
- settlement_limit: лимит +1 (макс 10)
- signal_range: SR +1 (без макс)
- decay_mitigation: DS -2 (макс 3)
- earthquake_mitigation: урон -2 (макс 3)
- beaver_damage_mitigation: урон -2 (макс 5)

### Формула очков
- Обычная клетка: 20 ходов × 5% × 10 = 1000 очков
- Усиленная клетка: 20 ходов × 5% × 15 = 1500 очков
- 30 плантаций постоянно работающих ≈ 30000-45000 очков/раунд
- Диверсия убивающая врага: столько очков сколько могло быть с клетки
- Убийство логова бобров: 10x очков клетки

### Порядок обработки сервера
1. Апгрейды → 2. Ремонт/Стройка → 3. Диверсии → 4. Атака логова → 5. Перенос ЦУ → 6. Атака бобров → 7. Деградация → 8. Урон недостроям → 9. Терраформация → 10. Респавн → 11. Катаклизмы

## Идеи для экспериментов

### Экспансия и стройка
1. Конвейер стройки: начинать новую до завершения терраформации текущей
2. Параллельная стройка: 2-3 плантации на одну цель (5+4+3=12 CS → 5 ходов)
3. Маршрут к усиленным: BFS к ближайшим (x%7, y%7) кластерам
4. Решётка вместо змейки: 2D сеть для устойчивости
5. Предсказание 100% терраформации: перенос ЦУ заранее

### Апгрейды
6. Все первые очки в settlement_limit
7. Динамический приоритет: менять по фазе игры (ранняя/средняя/поздняя)
8. Ранний signal_range для доступа к дальним целям

### Оптимизация
9. Экономия действий: не чинить то что скоро дотерраформирует
10. 100% утилизация: каждая плантация каждый ход что-то делает
11. Фокус на кластеры усиленных клеток

### Versus-специфичные
12. Диверсии на врагов у ценных клеток
13. Блокировка строек противника (строить на ту же клетку)
14. Защита ЦУ от диверсий (ремонт + буфер HP)
15. Агрессивная экспансия к территории противника
16. Разрушение цепочки связности врага (целиться в узкие места)
